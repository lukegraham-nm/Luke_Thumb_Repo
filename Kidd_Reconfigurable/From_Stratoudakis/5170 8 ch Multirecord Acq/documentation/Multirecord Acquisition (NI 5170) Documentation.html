Multirecord Acquisition (NI 5170)

The Multirecord Acquisition (NI 5170) sample project provides a basic starting point for acquiring data with NI PXIe-5170R/5171R/5172 reconfigurable oscilloscopes.

Features

  • Provides a session-based software interface to open an oscilloscope session; configure hardware, DSP, and trigger settings; and acquire data.
  • Allows the customizable FPGA VI to incorporate application-specific intellectual property (IP) to accelerate test execution speed.

System Requirements

  • 5170R/5171R Support
    • LabVIEW 2014 (32-bit) Full or Professional Development System or later.
    • NI PXI Express chassis with a NI 5170R/5171R installed. The chassis must either have an embedded PXIe controller or be connected to a host computer with a MXI-Express kit, such as the NI PXIe/PCIe-8375.
    • LabVIEW Instrument Design Libraries for Reconfigurable Oscilloscopes 14.0 or later.
    • (Optional) LabVIEW FPGA Module is required to view, open, and edit the included FPGA VI. If you do not have the LabVIEW FPGA module installed, you can still run this sample project; however, the Project Explorer window will show only the host VIs.
  • 5172 Support
    • LabVIEW 2016 (32-bit) Full or Professional Development System or later.
    • NI PXI Express chassis with a NI 5172 installed. The chassis must either have an embedded PXIe controller or be connected to a host computer with a MXI-Express kit, such as the NI PXIe/PCIe-8375.
    • LabVIEW Instrument Design Libraries for Reconfigurable Oscilloscopes 16.2.
    • (Optional) LabVIEW FPGA Module is required to view, open, and edit the included FPGA VI. If you do not have the LabVIEW FPGA module installed, you can still run this sample project; however, the Project Explorer window will show only the host VIs.

Note  Refer to the NI LabVIEW Instrument Design Libraries for Reconfigurable Oscilloscopes Readme for additional system requirements.

Overview

This sample project consists of four host VIs, which run on the host computer, and an FPGA VI, which runs on the FPGA within the device hardware. Each host VI instructs the FPGA VI on how to acquire data. The FPGA VI acquires data from the device input port.

You can run this sample project without installing the LabVIEW FPGA Module. however, you must install the LabVIEW FPGA Module to view, open, and edit the FPGA VI.

Host VIs

In the Project Explorer window, expand My Computer to see the following host VIs:

  • Configured Acquisition (Host).vi—Acquires a single record data continuously.
  • Configured Acquisition Trigger(Host).vi—Acquires data conditioned on the following trigger configurations:
    • Immediate: Acquisition is triggered as soon as the number of specified pre—trigger samples are acquired.
    • A (Edge Trigger, Level Trigger): Acquisition is triggered on source A when the criteria for selected trigger type for instance A are met.
    • B (Edge Trigger, Level Trigger): Acquisition is triggered on source B when the criteria for selected trigger type for instance B are met.
    • OR (Edge Trigger, Level Trigger): Acquisition is triggered on source A or B when the criteria for selected trigger type for instance A or B are met.
    • AND (Edge Trigger, Level Trigger): Acquisition is triggered on source A or B when the criteria for selected trigger type for instance A as well as B are met.
    • A THEN B (Edge Trigger, Level Trigger): Acquisition is triggered on source B when the criteria for selected trigger type for instance A followed by instance B are met.
    • B THEN A (Edge Trigger, Level Trigger): Acquisition is triggered on source A when the criteria for selected trigger type for instance B followed by instance A are met.
    • High Resolution (Edge Trigger, instance A only): Acquisition is triggered on source A with sub-sample accuracy.
  • External Clocking (Host).vi—Acquires data with external clock as the reference clock source.
  • Multiple Record Fetch (Host).vi—Acquires multiple records of data.

Instrument Design Libraries

If you have the LabVIEW FPGA Module installed, you can open the FPGA VI, which is visible in the Project Explorer window at <Device Name>»Multirecord Acquisition (FPGA).vi. This VI was created using several instrument design libraries. Each instrument design library consists of VIs that facilitate three aspects of programming the FPGA:

  • Register bus—The communication channel that performs the following tasks:
    • Allows direct reading and writing of the device registers
    • Arbitrates between instrument design libraries that require access to the same registers
  • FPGA resources—Memory, I/O, FIFOs, registers, and other resources that are dedicated to a particular instrument design library task.
  • Process—VIs that execute the task specific to a particular instrument design library.

You can see the instrument design libraries in the Project Explorer window by expanding the Dependencies»instr.lib project item. For example, the following screenshot shows some of the VIs in the Multirecord Acquisition instrument design library.

The Multirecord Acquisition instrument design library acquires data from the input connected to the device. The FPGA VI implements all aspects of this instrument design library (register bus, FPGA resources, process) in the following locations:

  • Register bus—niInstr FIFO Register Bus v1 FPGA.lvclass:Host Interface.vi is located in the Register Bus Loop section.
  • FPGA resources—niInstr Multirecord Acquisition Common v1 FPGA.lvclass:Create Resources DRAM U384x2 U768 Write Read.vi is located in the Create Acquisition Resources section.
  • Process—niInstr Multirecord Acquisition Common v1 FPGA.lvclass:Process.vi

Code in the Acquisition Loop section uses niInstr Multirecord Acquisition v1 FPGA.lvclass:Write I16 Wrapper.vi, which writes the acquired samples into memory, conditioned on the trigger.

Code in the Fetch Loop section uses niInstr Multirecord Acquisition v1 FPGA.lvclass:Retrieve Record U16 Wrapper.vi and niInstr Multirecord Acquisition v1 FPGA.lvclass:Write to Fetch DMA FIFO U16 Wrapper.vi to retrieve the data, write to the Fetch DMA FIFO and transmit it to the fetch VIs on the host.

In addition to the Multirecord Acquisition instrument design library, this sample project demonstrates the usage of the following instrument design libraries:

  • Configuration—Configures the device hardware.
  • Data Trigger—Generates triggers based on the input signal.
  • DSP—Processes data using common digital signal processor (DSP) algorithms.
  • Register Bus—Sends register read and write instructions from your application on the host to the FPGA.

When you are programming the NI 5170R/5171R/5172, access the instrument design libraries in the following palette locations:

  • Host VIs:
    Functions»FPGA Interface»Software-Designed Instruments»Oscilloscopes»NI PXIe-5170R Functions»FPGA Interface»Software-Designed Instruments»Oscilloscopes»NI PXIe-5171R Functions»FPGA Interface»Software-Designed Instruments»Oscilloscopes»NI PXIe-5172
  • FPGA VIs:
    Functions»NI PXIe-5170R Functions»NI PXIe-5171R Functions»NI PXIe-5172

FPGA VI Design

The FPGA VI acquires data from the input port using the Multirecord Acquisition instrument design library to acquire data from the input port and store it in onboard memory.

The following five loops run in parallel in the FPGA VI:

  • Register Bus
    • Configures the FPGA communication channels for data acquisition using an instance of the Register Bus instrument design library.
  • Acquisition Loop
    • Reads acquired data from analog-to-digital converters (ADCs) and applies digital correction.
    • Processes this data using the DSP instrument design library.
    • Using the data, generates triggers which are used by the Multirecord Acquisition instrument design library.
    • Stores data in DRAM using the Multirecord Acquisition instrument design library.
  • Fetch Loop
    • Retrieves data stored in DRAM by the acquisition loop using the Multirecord Acquisition instrument design library.
    • Sends this data to the host computer through a DMA channel.

Running this Sample Project

  1. Ensure all hardware is set up and visible in NI Measurement & Automation Explorer (MAX). Record the resource names of all NI 5170R/5171R/5172 devices; you will need these names later.
  2. Connect an arbitrary waveform generator or other appropriate equipment to the input port using high-quality X-to-SMA cables.
  3. In the Project Explorer window, open a host VI, such as Configured Acquisition Trigger(Host).vi.
  4. In the RIO Device pull-down menu, specify the resource name of the device that you identified in step 1.
  5. Click Run. The device begins acquiring.
  6. Click Stop.
  7. Experiment with different values for the front panel controls and click Run. You also can open one of the other host VIs.

    Note  Error handling in the instrument design libraries differs from error handling in NI-SCOPE driver software. Less error checking occurs in the instrument design libraries. For example, entering a value that is out of range may not result in an error. You can create custom error codes using the sample project error handlers, as described in the following section.

Modifying this Sample Project

Device Session.lvclass is an aggregation of the instrument design libraries and can be customized to suit your application requirements. The following sections describe common modifications you can make to the FPGA VI. You must install the LabVIEW FPGA Module to access the FPGA VI. With this module installed, launch the FPGA VI from the Project Explorer window by opening My Computer»<Device Name>»Multirecord Acquisition (FPGA).vi. This code is user-owned, namespaced by the created project's name, and can be customized.

Adding Custom Triggering

You can add custom triggering code in the Acquisition Loop section. In this section, triggering occurs after the DSP block. Trigger.vi is designed to generate triggers using two-channel source combinations. This VI can be modified to accept more than two channels. You can also add a custom trigger type not already provided by the Data Trigger instrument design library. You can also add additional trigger combination logic. This will require the Trigger Combination Type enum to be updated. To do so, open this enum from the Project Explorer window by selecting My Computer»FPGA Interface Typedefinitions.lvlib»Trigger Combination Type.ctl and add the new combination logic case to it. Any changes to the trigger logic can affect the delay in data path on the FPGA.

Adding Custom DSP Code

You can add custom DSP code in the appropriate locations in the Acquisition Loop section. Place custom DSP code after the Digital Correction VI, which contains code for the digital correction filter. You can also add custom DSP code in the Fetch Loop section, between the Retrieve Record VI and the Write to Fetch DMA FIFO VI. Any changes to the DSP logic can affect the delay in data path on the FPGA.

Using Digital I/O

Digital I/O lines can be added as follows:

  • You can use either PFI or PXI lines for digital I/O.
    • To add a PXI line from the Project Explorer window, right-click the device name and select New»FPGA I/O, then select the desired PXI line from the New FPGA I/O window.
    • PFI lines are already present in the Project Explorer window under My Computer»<Device Name>»Resources»Trigger and Digital»<PFI Line>.
  • To create an FPGA I/O Node for a line, drag the line to the block diagram of the FPGA VI.

Note  For more information about using PXI and PFI lines, refer to the NI Reconfigurable Oscilloscopes Help.

NI recommends designing and testing digital I/O code before integrating it with your main application code.

Adding Custom Error Codes

You can add custom error codes for Device Session.lvclass. Complete the following steps to add a custom error code:

  1. From the Project Explorer window, select View»Functions Palette.
  2. In the Functions palette, select Programming»Dialog & User Interface. Drag an Error Ring to the VI to add it.
  3. Click on the Error Ring and select Custom Error Code from the Error Code Range drop-down menu. Enter an appropriate Error Code and Error Description. Make sure that the Error Code does not overwrite an existing code.
